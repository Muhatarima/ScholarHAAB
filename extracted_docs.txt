1. The Vision
ScholarHAAB is not a chatbot. It is not a search engine. It is not a directory of scholarships.
It is the knowledgeable friend that most Bangladeshi students never had — the one who actually knows how Chevening applications work, what a strong SOP looks like, how to approach a professor for PhD funding, and how to solve a Cambridge A Level calculus problem step by step at 2 AM before an exam.
Core belief: The difference between a student who gets a scholarship and one who does not is often not ability — it is access to the right guidance at the right time. ScholarHAAB closes that gap.
1.1 — What ScholarHAAB Is
A scholarship intelligence system that matches students to real funding opportunities based on their actual profile
A document evaluator that reads SOPs, recommendation letters, CVs, essays, and transcripts and tells students exactly what is weak and how to fix it
An O and A Level QBank engine that solves Cambridge and Edexcel past papers with full working, explanations, and adaptive tutoring
A study abroad consultant that understands Bangladesh-specific constraints: grading systems, university reputation, family pressure, financial reality, and visa risk
1.2 — What ScholarHAAB Is NOT
Not a generic AI assistant that gives vague advice
Not a scholarship listing website
Not a service that overpromises visa success or admission guarantees
Not a product that wastes a student's time with irrelevant information

Warm, human tone — not customer service, not robotic, not dramatic

2. Security Requirements
2.1 — Authentication
CRITICAL: Max 3 active device sessions per user account. If a 4th device logs in, the oldest session is invalidated automatically.
Email + password authentication via Supabase Auth
OTP verification required on every new device login
OTP expires in 10 minutes — single use only
Session tokens stored in httpOnly cookies — never in localStorage
JWT tokens expire in 1 hour, refresh tokens in 7 days
Max 3 concurrent sessions per user — tracked in user_sessions table
Automatic session invalidation on password change
2.2 — Device Session Management
-- user_sessions table
CREATE TABLE user_sessions (
  id          UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id     UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  device_info TEXT,
  ip_address  TEXT,
  created_at  TIMESTAMP DEFAULT NOW(),
  last_active TIMESTAMP DEFAULT NOW(),
  is_active   BOOLEAN DEFAULT TRUE
);
-- Enforce max 3 sessions via trigger
CREATE OR REPLACE FUNCTION enforce_session_limit()
RETURNS TRIGGER AS $$
BEGIN
  IF (SELECT COUNT(*) FROM user_sessions
      WHERE user_id = NEW.user_id AND is_active = TRUE) >= 3 THEN
    UPDATE user_sessions SET is_active = FALSE
    WHERE user_id = NEW.user_id AND is_active = TRUE
    AND created_at = (SELECT MIN(created_at) FROM user_sessions
      WHERE user_id = NEW.user_id AND is_active = TRUE);
  END IF;
  RETURN NEW;
END; $$ LANGUAGE plpgsql;
2.3 — API Security
All API routes require valid Supabase JWT — no public endpoints except /auth
Rate limiting: 60 requests/minute per user on /api/chat
Rate limiting: 10 requests/minute on /api/evaluate-sop
Input sanitization on all user inputs before AI call
Max input length: 2000 characters for chat, 5000 for SOP evaluation
All Gemini/OpenAI API calls happen server-side only — keys never exposed to client
CORS restricted to scholorhaab.com domain only in production
Supabase Row Level Security (RLS) enabled on all tables

4. Payment System — SSLCommerz
4.1 — Payment Flow
User clicks Upgrade → selects Pro or Premium
Frontend calls /api/payment/initiate with tier + user_id
Backend creates SSLCommerz session with order details
User redirected to SSLCommerz payment page
User completes payment (bKash, Nagad, card, etc.)
SSLCommerz sends IPN (webhook) to /api/payment/ipn
Backend validates IPN signature — MUST validate before any action
On success: update subscriptions table, update profiles.tier
User redirected to success page, new tier active immediately
4.2 — IPN Validation (CRITICAL)
NEVER update subscription status without validating IPN signature. Always use SSLCommerz validation API before trusting any payment callback.
// app/api/payment/ipn/route.ts
export async function POST(req: Request) {
  const data = await req.formData()
  const val_id = data.get('val_id')
  // Step 1: Validate with SSLCommerz
  const validation = await fetch(
    `https://securepay.sslcommerz.com/validator/api/validationserverAPI.php?val_id=${val_id}&store_id=${STORE_ID}&store_passwd=${STORE_PASS}&format=json`
  )
  const result = await validation.json()
  if (result.status !== 'VALID' && result.status !== 'VALIDATED') {
    return Response.json({ error: 'Invalid payment' }, { status: 400 })
  }
  // Step 2: Only now update database
  await activateSubscription(result.value_a, result.value_b)
  return Response.json({ success: true })
}
